For server:
Users can create public or private servers.
Public servers: anyone can join/leave, no permissions needed.
Private servers: creator is admin. 
Only admin can delete server and channels (both public and private). 
Admin can invite/kick users by username, and grant invite/kick permission to others, delete server, give permissions to create text channels.
Any user can leave a private/public server.
Only admin can grant permissions; no one else. Currently not supporitng multiple admins.
If admin leaves, he has to transfer admin permissions to another user before leaving, otherwise the server will be deleted.
Anyone in the server has access to all channels in that server, but only users with permissions can create channels.
User Roles in servers: Admin, Invite Permission in Server, Kick Permission from Server, Create Text Channel Permission, Membership (no permissions).
When a user is deleted, their memberships and invites are removed.
If a server is deleted, all channels and memberships and invites for that server are removed.

SQL Relationships Overview
1. users
Primary Table: All other tables reference users.
Cascade: Deleting a user removes their memberships, roles, invites, friendships, blocks, and sets their messages’ sender to NULL.
2. friends
Relationship: Many-to-many between users.
Cascade: If either user is deleted, the friendship is deleted.
Constraint: No self-friendship.
3. blocks
Relationship: Many-to-many between users.
Cascade: If either user is deleted, the block is deleted.
Constraint: No self-blocking.
Trigger: Blocking a user deletes any friendship between them.
4. servers
Relationship: Each server has an owner (user).
Cascade: If the owner is deleted, ownerID is set to NULL.
Cascade: Deleting a server removes all its channels, roles, invites, memberships, and messages.
5. server_members
Relationship: Many-to-many between users and servers.
Cascade: Deleting a user or server removes the membership.
6. server_roles
Relationship: Each role belongs to a server.
Cascade: Deleting a server removes its roles.
7. server_member_roles
Relationship: Many-to-many between server_members and server_roles.
Cascade: Deleting a membership or role removes the assignment.
8. server_invites
Relationship: Invites link users and servers, and reference the inviter (must be a member).
Cascade: Deleting a user or server removes related invites.
9. channels
Relationship: Each channel belongs to a server and may have a creator (user).
Cascade: Deleting a server removes its channels.
Cascade: Deleting the creator sets created_by to NULL.
10. channel_role_permissions
Relationship: Links channels and roles for permissions.
Cascade: Deleting a channel or role removes related permissions.
11. direct_conversations
Relationship: Each conversation is between two users.
Cascade: Deleting either user removes the conversation.
Constraint: No self-conversation.
12. messages
Relationship: Each message belongs to either a channel or a direct conversation, and has a sender (user).
Cascade: Deleting a channel or conversation removes related messages.
Cascade: Deleting the sender sets senderID to NULL.
Constraint: Message must belong to either a channel or a conversation, not both.
13. Triggers
trg_servers_after_insert_bootstrap: When a server is created, default roles and memberships are set up.
trg_server_members_after_insert_assign_default_role: When a member joins, they get the default role.
trg_blocks_after_insert_remove_friendship: Blocking a user deletes any friendship between them.
